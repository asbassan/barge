# BARGE v0.1.0 — Bug Fixes

All bugs encountered and fixed during v0.1.0 development and testing.

---

## BUG-01 — `barge build -f` unknown shorthand flag

**Symptom**
```
Error: unknown shorthand flag: 'f' in -f
```

**Root cause**
The `--file` flag in `newBuildCmd()` was registered with `StringVar` instead of `StringVarP`. `StringVar` registers the long form only; `StringVarP` registers both `--file` and `-f`.

**Fix**
`cmd/barge/main.go` — changed `StringVar` → `StringVarP` with shorthand `"f"`:
```go
cmd.Flags().StringVarP(&bargefilePath, "file", "f", "Bargefile", "Path to the Bargefile or Dockerfile")
```

## BUG-02 — `.` treated as unknown command

**Symptom**
```
barge.exe -f Bargefile -t tessa1:v1 .
Error: unknown command '.' for 'barge'
```

**Root cause**
The `build` subcommand was omitted. Cobra parsed `.` as a subcommand of the root `barge` command.

**Fix**
User error — correct syntax is:
```
barge build -f Bargefile -t tessa1:v1 .
```

## BUG-03 — Docker Hub image reference parsed as URL

**Symptom**
```
Error: FROM: failed to pull "python:3.11-windowsservercore-ltsc2022"
  Original error: parse "dummy://python:3.11-windowsservercore-ltsc2022": invalid port ":3.11-windowsservercore-ltsc2022" after host
```

**Root cause**
containerd's reference parser treats short Docker Hub names (e.g. `python:3.11-slim`) as URLs. Without a registry prefix, `python` is parsed as the hostname and `:3.11-...` as the port number.

**Fix**
`internal/client/images.go` — added `normalizeRef()` which expands short references before passing to containerd:
- `python:3.11-slim` → `docker.io/library/python:3.11-slim`
- `myorg/myapp:latest` → `docker.io/myorg/myapp:latest`
- `mcr.microsoft.com/windows/nanoserver:ltsc2022` → unchanged

Applied in `Pull()` and `GetImage()`.

## BUG-04 — Detached container fails with `scheme must be 'binary', got: 'file'`

**Symptom**
```
Error: cannot start build container: failed to create task: failed to create shim task: scheme must be 'binary', got: 'file': unknown
```

**Root cause**
`cio.LogFile(path)` passes a `file://` URI to the container shim for log redirection. On Linux, containerd shims accept `file://`. `containerd-shim-runhcs-v1` (Windows) only accepts `binary://` or null — it hard-rejects `file://`.

**Fix**
`internal/client/run.go` — changed detached containers to use `cio.NullIO` instead of `cio.LogFile`. The build keep-alive container (`ping -t 127.0.0.1`) does not need its output captured. Output from `RUN` steps is still visible via `Exec` which uses host stdio.

## BUG-05 — COPY single file produces invalid ZIP entry `.`

**Symptom**
```
ExpandArchiveHelper : Can not process invalid archive entry '.'.
```
Only affected single-file COPY sources (e.g. `COPY pyproject.toml ./`). Directory sources were unaffected.

**Root cause**
`fileserver.go`'s `zipDir()` used `filepath.Walk` for both files and directories. When the source is a single file, `filepath.Rel(file, file)` returns `.`, so the ZIP entry was named `.` — which `Expand-Archive` rejects as an invalid path.

**Fix**
`internal/build/fileserver.go` — `zipDir()` now checks `os.Stat` first:
- If source is a **file**: zip it under `filepath.Base(src)` (e.g. `pyproject.toml`)
- If source is a **directory**: zip all files with relative paths as before

## BUG-06 — WORKDIR directory not created inside the container

**Symptom**
```
RUN python -m pip install ...
The system cannot find the path specified.
```
`cd /d C:\app` failed because `C:\app` was never created.

**Root cause**
The `WORKDIR` instruction recorded the path in `buildState.workDir` but never actually created the directory inside the running build container. Subsequent `RUN` steps prepend `cd /d <workDir>` which fails if the directory does not exist.

**Fix**
`internal/build/builder.go` — `WORKDIR` processing now execs `New-Item -ItemType Directory -Force` inside the container immediately after recording the path:
```go
mkCmd := fmt.Sprintf("New-Item -ItemType Directory -Force '%s' | Out-Null", winDir)
_ = b.cl.Exec(ctx, client.ExecOptions{
    ContainerID: buildID,
    Args:        []string{"powershell.exe", "-NonInteractive", "-Command", mkCmd},
})
```

## BUG-07 — COPY relative destination not resolved against WORKDIR

**Symptom**
```
ExpandArchiveHelper : Can not process invalid archive entry '.'.
```
`COPY pyproject.toml ./` passed `./` literally to `Expand-Archive -DestinationPath './'` inside the container.

**Root cause**
`execCopy` called `toWindowsPath("./")` which returned `./` unchanged (not an absolute path), so `Expand-Archive` received a relative path it could not resolve.

**Fix**
`internal/build/builder.go` — added `resolveCopyDst(dst, workDir)` which resolves relative destinations against the current WORKDIR before passing to the container:
- `./` with `WORKDIR C:\app` → `C:\app`
- `./src` with `WORKDIR C:\app` → `C:\app\src`
- `C:\absolute` → unchanged

## BUG-09 — Locally built images not found by `barge run`

**Symptom**
```
Error: image "docker.io/library/tessa1:v1" not found locally
  Pull it first: barge pull docker.io/library/tessa1:v1
```

**Root cause**
`GetImage` called `normalizeRef` unconditionally, converting `tessa1:v1` to `docker.io/library/tessa1:v1`. Pulled images are stored under the normalized name, but locally built images are stored under the name the user gave them (`tessa1:v1`), so the lookup failed.

**Fix**
`internal/client/images.go` — `GetImage` now tries the normalized ref first, then falls back to the original ref if not found:
```go
if img, err := cl.c.GetImage(ctx, normalized); err == nil { return img, nil }
if img, err := cl.c.GetImage(ctx, ref); err == nil { return img, nil }
```

## BUG-10 — `barge run` fails with `content digest sha256:...: not found` after successful build

**Symptom**
```
Error: failed to create container "tessa1":
  content digest sha256:0df95203fcea43722a838ec3b48bc2805965e81b807acd1a6b66d755cde89f28: not found
```
Image appears in `barge images` with correct size (2.19 GB); the diff layer IS stored in the content store.

**Root cause**
`CommitContainer` wrote all OCI blobs (diff layer, config, manifest) and created the image record, but never called `img.Unpack()`. When `barge run` calls `WithNewSnapshot`, containerd computes the chain ID from DiffIDs and calls `snapshotter.Prepare(ctx, id, chainID)`. Since the new layer was never unpacked into a VHD snapshot, the snapshotter tries to read the blob and fails with "content not found."

**Fix**
`internal/client/commit.go` — after creating the image record, unpack the image to convert the OCI layer tarballs into VHD snapshots:
```go
newImg := containerd.NewImage(cl.c, newImgRecord)
if err := newImg.Unpack(ctx, windowsSnapshotter); err != nil {
    return fmt.Errorf("cannot unpack new image: %w", err)
}
```

## BUG-14 — `barge run` swallows container command flags (e.g. `ping -t`)

**Symptom**
```
barge run -d --name tessa1 tessa1:v1 ping -t 127.0.0.1
Error: unknown shorthand flag: 't' in -t
```
Any flag passed to the container command (e.g. `-t`, `-n`, `-c`) is parsed as a barge flag instead.

**Root cause**
Cobra's default interspersed flag parsing applies to all arguments regardless of position. After the barge flags (`-d`, `--name`) and image name, cobra continued scanning for barge flags and hit `-t`.

**Fix**
`cmd/barge/main.go` — added `cmd.Flags().SetInterspersed(false)` to `newRunCmd()` so cobra stops parsing flags once the first non-flag positional argument (the image name) is encountered. Everything after the image is now treated as container command + args:
```go
cmd.Flags().SetInterspersed(false)
```
Also removed the `--` separator requirement from the args parsing — `opts.Args = args[1:]` is now sufficient.

## BUG-13 — Re-running `barge run --name X` fails with `snapshot "X": already exists`

**Symptom**
```
Error: failed to create container "tessa1": snapshot "tessa1": already exists
```
Happens when `barge run --name tessa1 tessa1:v1` is run a second time after a previous foreground run of the same name has already exited.

**Root cause**
Foreground `barge run` without `--rm` deleted the task but left the container record and its snapshot in the store. The next run with the same name tried to create a new snapshot with the same key, which already existed.

**Fix**
`internal/client/run.go` — foreground runs now always clean up the container (and its snapshot) on exit, regardless of the `--rm` flag. Detached (`-d`) containers still persist until explicitly removed. `--rm` is accepted but is now a no-op for foreground runs (kept for Docker muscle-memory compatibility):
```go
// Always remove foreground containers on exit.
network.DeleteEndpoint(endpointID)
container.Delete(ctx, containerd.WithSnapshotCleanup)
```

## BUG-11 — `barge build` commit fails with `wrong diff id calculated on extraction`

**Symptom**
```
time="..." level=info msg="apply failure, attempting cleanup" error="wrong diff id calculated on extraction \"sha256:5958a0c7...\""
Error: commit: cannot unpack new image: wrong diff id calculated on extraction "sha256:5958a0c7..."
```
Appeared after BUG-10 fix (adding `Unpack`). The `Unpack` call itself fails because the DiffID stored in the image config does not match the SHA256 containerd computes during extraction.

**Root cause**
The OCI DiffID must be the SHA256 of the **uncompressed** layer tar. The Windows diff plugin (`containerd-shim-runhcs-v1`) does not set the `containerd.io/uncompressed` annotation on the descriptor returned by `DiffService().Compare()`. The original fallback was to use `layerDesc.Digest` (the SHA256 of the **compressed** blob) as the DiffID — which is wrong. When `Unpack` decompresses the layer and hashes the result, it gets a different digest and rejects the import.

**Fix**
`internal/client/commit.go` — added `uncompressedDigest()` helper that reads the gzip blob from the content store, decompresses it on the fly, and returns the SHA256 of the raw tar:
```go
func uncompressedDigest(ctx context.Context, cs content.Store, desc ocispec.Descriptor) (digest.Digest, error) {
    ra, _ := cs.ReaderAt(ctx, desc)
    defer ra.Close()
    gr, _ := gzip.NewReader(io.NewSectionReader(ra, 0, desc.Size))
    defer gr.Close()
    digester := digest.SHA256.Digester()
    io.Copy(digester.Hash(), gr)
    return digester.Digest(), nil
}
```
Used as the fallback when the annotation is absent.

## BUG-12 — `barge run` fails with `content digest sha256:...: not found` after successful build (GC eviction)

**Symptom**
```
Error: failed to create container "tessa1":
  content digest sha256:6f423db150e1aad8eabed9661f4c8bffe5b82e7ffd1a1b6f2c0fbea0c6732d52: not found
```
Build finishes with "Successfully built tessa1:v1" (Unpack succeeds). `barge run` immediately fails. The missing digest is the image config blob, not the layer or manifest.

**Root cause**
Two compounding issues:
1. **GC race — Delete+Create window**: `CommitContainer` deleted the old image record before creating the new one. During that brief gap, no image record anchored the newly written config and manifest blobs. If containerd's background GC ran during that window, it could evict both blobs.
2. **No permanent GC root labels**: The config and manifest blobs were written only within a lease. Once the lease expired (function returned), the sole anchor was the image record — but issue #1 created a window where that anchor was absent.

**Fix**
`internal/client/commit.go` — two changes:

1. After writing each blob (layer, config, manifest), set `containerd.io/gc.root: barge` label via `cs.Update`. This makes them unconditional GC roots so they survive any lease/record gap:
```go
gcLabel := map[string]string{"containerd.io/gc.root": "barge"}
for _, d := range []digest.Digest{layerDesc.Digest, configDesc.Digest, manifestDesc.Digest} {
    cs.Update(ctx, content.Info{Digest: d, Labels: gcLabel}, "labels.containerd.io/gc.root")
}
```

2. Replaced Delete→Create with atomic Create→Update-if-exists, so the image record always exists while the new blobs are live:
```go
newImgRecord, err := imgSvc.Create(ctx, images.Image{Name: newRef, Target: manifestDesc})
if errdefs.IsAlreadyExists(err) {
    newImgRecord, err = imgSvc.Update(ctx, images.Image{Name: newRef, Target: manifestDesc})
}
```

## BUG-15 — Re-running `barge run -d --name X` fails with `snapshot "X": already exists` (detached container)

**Symptom**
```
barge run -d --name tessa1 tessa1:v1
# container runs and exits (status: stopped)
barge run -d --name tessa1 tessa1:v1 ping -t 127.0.0.1
Error: failed to create container "tessa1": snapshot "tessa1": already exists
```
Affects detached (`-d`) containers that have exited but not been explicitly removed. The user gets a raw containerd error with no remedy.

**Root cause**
Detached containers persist until `barge rm` — this is intentional (matches Docker semantics). However, when a user re-runs with `--name X` without removing first, `WithNewSnapshot(id, img)` tries to create a snapshot with the same key, which fails because the old one still exists. The error from containerd is not actionable.

**Fix**
`internal/client/run.go` — added an existence check at the top of `Run()`. If `cl.c.LoadContainer(ctx, id)` succeeds, return immediately with a helpful message:
```
container named "tessa1" already exists

  Remove it first:
    barge rm tessa1
  Or force-remove:
    barge rm -f tessa1
```

## BUG-08 — Commit fails with `snapshot not active or view: failed precondition` / `number of mounts should always be 1`

**Symptom**
```
Error: commit: cannot get parent snapshot mounts: failed to get snapshot mount:
  requested snapshot barge/52/sha256:a99842a... not active or view: failed precondition
```

**Root cause**
`CommitContainer` called `snapshotter.Mounts(ctx, snapInfo.Parent)` to get the lower (parent image) mounts for the diff comparison. On Windows, the snapshotter stores layers as VHD files in a chain. Calling `Mounts()` on a **committed** image-layer snapshot is not supported — only active (writable) or view (read-only) snapshots can be mounted. The parent is a committed image layer, so the call fails.

This is a fundamental Linux/Windows difference: Linux overlayfs requires explicit lower mount paths for diff computation; the Windows VHD diff plugin derives the parent chain from the VHD metadata internally.

**Fix**
`internal/client/commit.go` — two-stage fix:

1. Passing `nil` lower resolved the first error but produced a second: `number of mounts should always be 1 for Windows layers`. The Windows diff plugin requires exactly 1 lower mount.

2. Committed snapshots can't be mounted via `Mounts()` directly, but a **view** snapshot (read-only overlay) of the same parent CAN be mounted. Create a temporary view, use its mounts as lower, then remove the view after the diff:

```go
viewKey := containerID + "-lower-view"
lower, err = snapshotter.View(ctx, viewKey, snapInfo.Parent, ...)
defer snapshotter.Remove(ctx, viewKey)
layerDesc, err := cl.c.DiffService().Compare(ctx, lower, upper, ...)
```
